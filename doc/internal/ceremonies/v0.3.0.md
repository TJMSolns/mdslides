# Domain Ceremonies - v0.3.0 (Image Asset Management)

**Date:** December 22, 2025
**Participants:** Development Team
**Facilitator:** Domain Expert

## Session Overview

This document captures the Event Storming and Domain Ceremonies for **v0.3.0**, which adds automatic image asset copying to MDSlides, improving the user workflow for presentations with images.

## Key Decisions Made

### User Requirements (December 22, 2025):
1. **Automatic Image Copying**: When generating HTML, automatically copy referenced local image files to output directory
2. **Path Rewriting**: Update image paths in HTML to match copied location
3. **Directory Structure Preservation**: Maintain relative directory structure when copying images
4. **Validation**: Error if referenced image file doesn't exist
5. **User Control**: Provide CLI flags to enable/disable copying behavior
6. **Portability**: Generated presentations should be self-contained and portable

### Technical Decisions:
- **CLI Flags**: `--copy-images` (default), `--no-copy-images`, `--images-dir <path>`
- **Path Detection**: Distinguish local paths from external URLs and data URLs
- **Error Handling**: Fail generation if referenced local image doesn't exist
- **File I/O**: Use `java.nio.file.Path` for cross-platform compatibility

## Changes from v0.2.0

**Current state (v0.2.0):**
- Images render with paths provided in markdown
- User manually copies images alongside HTML output
- Easy to forget images, resulting in broken presentations
- Distribution requires separate image files

**Target state (v0.3.0):**
- Images automatically copied to output directory
- Paths rewritten in HTML to match copied location
- Validation ensures all referenced images exist
- Single directory contains HTML + images (portable)
- User can opt-out with `--no-copy-images`

## Event Storming Session

### Domain Events (Time-Ordered)

1. **PresentationSourceParsed**
   - Markdown file read and parsed
   - Image references extracted from slides
   - Triggers: ImageReferencesDetected

2. **ImageReferencesDetected**
   - List of image URLs from markdown `![alt](url)` syntax
   - Distinguished: local paths vs URLs vs data URLs
   - Triggers: LocalImagePathsResolved

3. **LocalImagePathsResolved**
   - Relative paths resolved to absolute filesystem paths
   - Based on source markdown file location
   - Triggers: ImageFilesValidated

4. **ImageFilesValidated**
   - Each referenced local image file checked for existence
   - File permissions checked (readable)
   - Triggers: ImageFilesCopied (success) or ImageValidationFailed (error)

5. **ImageFilesCopied**
   - Image files copied to output directory
   - Directory structure preserved (relative paths maintained)
   - Triggers: ImagePathsRewritten

6. **ImagePathsRewritten**
   - HTML `<img>` tags updated with new paths
   - Paths relative to generated HTML file
   - Triggers: PresentationGenerated

7. **PresentationGenerated**
   - HTML file written with corrected image paths
   - Output directory contains HTML + images
   - Presentation is self-contained and portable

### Commands

- **ParsePresentationSource**: Read markdown file, extract image references
- **DetectImageReferences**: Identify all `![alt](url)` in slides
- **ResolveLocalPaths**: Convert relative paths to absolute based on source location
- **ValidateImageFiles**: Check all referenced images exist and are readable
- **CopyImageAssets**: Copy image files to output directory, preserving structure
- **RewriteImagePaths**: Update HTML with new relative paths
- **GeneratePresentation**: Write final HTML with corrected paths

### Aggregates

#### ImageAsset (New Value Object)
- `originalPath: String` - Path as specified in markdown
- `resolvedPath: Path` - Absolute filesystem path
- `destinationPath: String` - Relative path in output directory
- `imageType: ImageType` - Local | External | DataURL

#### ImageCopyManifest (New Aggregate)
- `imageAssets: List[ImageAsset]` - All images to copy
- `sourceDirectory: Path` - Base directory for resolving relative paths
- `outputDirectory: Path` - Destination for copied images
- `validated(): Either[ValidationError, ImageCopyManifest]` - Verify all files exist

### Read Models

#### ImageReferenceList
- List of all image URLs found in presentation
- Used for validation and copying

#### ImageCopyReport
- List of successfully copied images
- File sizes, source/destination paths
- Total size of image payload

## User Stories for v0.3.0

### US-006: Image Asset Copying
**As a** presentation author
**I want** MDSlides to automatically copy referenced image files to the output directory
**So that** I can distribute a single self-contained directory without worrying about broken image paths

**Acceptance Criteria:**

#### Must Have
1. **Local file detection**:
   - Detect relative paths: `./images/logo.png`, `images/logo.png`
   - Detect absolute local paths: `/home/user/images/logo.png`
   - Skip external URLs: `https://example.com/logo.png`
   - Skip data URLs: `data:image/...`

2. **Image copying**:
   - Copy images to output directory structure
   - Preserve relative directory structure
   - Handle duplicate filenames (error with clear message)

3. **Path rewriting**:
   - Rewrite paths in HTML to match copied location
   - Maintain relative paths from HTML to images

4. **Error handling**:
   - Error if referenced image file doesn't exist
   - Error if image file can't be read (permissions)
   - Error if image can't be copied (disk space, permissions)

5. **CLI flags**:
   ```bash
   # Copy images (new default behavior)
   mdslides slides.md output.html --copy-images

   # Don't copy images (legacy behavior)
   mdslides slides.md output.html --no-copy-images

   # Specify custom output directory for images
   mdslides slides.md output.html --images-dir assets/images
   ```

#### Should Have
6. **Image validation**:
   - Verify image files are valid (magic number check)
   - Warn if image file is very large (>5MB)
   - Warn if total image payload is excessive (>20MB)

7. **Progress reporting**:
   ```
   Copying images...
   ✓ Copied images/logo.svg (4.2 KB)
   ✓ Copied images/diagram.png (128 KB)
   ✓ Copied 2 images (132.2 KB total)
   ```

#### Could Have
8. **Image optimization** (future):
   - Resize large images
   - Convert to optimal formats
   - Compress without quality loss

9. **Bundle mode** (future):
   - Embed all images as data URLs
   - Single-file HTML output

**Examples:**

**Example 1: Basic Usage**

Input structure:
```
project/
├── slides.md
└── assets/
    └── logo.svg
```

slides.md:
```markdown
![Logo](assets/logo.svg)
```

Command:
```bash
mdslides slides.md output/presentation.html --copy-images
```

Output structure:
```
project/
├── slides.md
├── assets/
│   └── logo.svg
└── output/
    ├── presentation.html
    └── assets/
        └── logo.svg  # Copied
```

**Example 2: Custom Images Directory**

Command:
```bash
mdslides slides.md output.html --copy-images --images-dir img
```

Output structure:
```
project/
├── slides.md
├── output.html
└── img/         # Custom directory
    └── logo.svg
```

**Example 3: No Copying (Legacy)**

Command:
```bash
mdslides slides.md output.html --no-copy-images
```

User must manually ensure images are in the correct location.

**Alternative Approaches Considered:**

1. **Always embed as data URLs** - Rejected: Large file sizes, harder to edit images, no browser caching
2. **Symbolic links** - Rejected: Breaks portability, doesn't work on Windows without admin
3. **Package as ZIP** - Deferred: Could be future enhancement, requires extraction step

## Technical Design

### Domain Layer (No Changes Required)

**ContentImage** already exists as value object. No domain changes needed.

### Infrastructure Layer

**New Component:** `ImageAssetCopier`

```scala
package com.tjmsolutions.mdslides.infrastructure.assets

import java.nio.file.Path
import cats.effect.IO

/**
 * Copies image assets referenced in slides to output directory.
 *
 * Anticorruption Layer for file I/O operations.
 */
object ImageAssetCopier:

  /**
   * Copy images to output directory.
   *
   * @param images List of image URLs from slides
   * @param sourceDir Base directory for resolving relative paths
   * @param outputDir Destination directory for copied images
   * @return IO effect that produces list of copied files or error
   */
  def copyImages(
    images: List[String],
    sourceDir: Path,
    outputDir: Path
  ): IO[Either[String, List[CopiedImage]]]

  /**
   * Detect if URL is a local file path.
   */
  def isLocalPath(url: String): Boolean

  /**
   * Resolve relative path to absolute path.
   */
  def resolvePath(url: String, baseDir: Path): Either[String, Path]

case class CopiedImage(
  originalPath: String,
  destinationPath: String,
  sizeBytes: Long
)
```

### CLI Layer

**Changes to Main.scala:**

```scala
val copyImages = Opts.flag("copy-images", help = "Copy image files to output directory (default: true)").orFalse
```

## Architecture Decision Records

### ADR-012: Image Asset Copying Strategy

**Status:** Proposed

**Context:**
- v0.2.0 added image embedding, but users must manually copy images
- Easy to forget images when distributing presentations
- Broken image paths are common pain point

**Decision:**
- Automatically copy referenced local image files to output directory
- Preserve relative directory structure
- Provide CLI flags to control behavior

**Consequences:**
- **Positive:** Self-contained presentations, better user experience
- **Positive:** Validation catches missing images before distribution
- **Negative:** Slightly longer build time (file I/O)
- **Negative:** Duplicate disk space for images (mitigated: only for output)

**Alternatives Considered:**
- Symlinks: Rejected (not portable)
- Data URL embedding: Rejected (large files, no caching)
- ZIP packaging: Deferred (future enhancement)

## Product Decision Records

### PDR-009: Image Copy Default Behavior

**Status:** Proposed

**Decision:**
- Image copying is **enabled by default** (`--copy-images` implicit)
- Users can opt-out with `--no-copy-images`
- Custom output directory via `--images-dir`

**Rationale:**
- Most users want portable presentations
- Safer default (prevents broken images)
- Power users can opt-out if needed

**Migration:**
- No breaking change (new feature, defaults to helpful behavior)

### PDR-010: Image Size Limits

**Status:** Proposed

**Decision:**
- **Warn** (not fail) if single image >5MB
- **Warn** (not fail) if total image payload >20MB
- Users can override warnings

**Rationale:**
- Large images slow page load
- But blocking builds is too aggressive
- Users know their constraints (conference wifi vs local presentation)

## Risks and Mitigation

### Risk 1: Overwriting existing files
**Mitigation:** Check if destination file exists, error with clear message

### Risk 2: Large images causing slow builds
**Mitigation:** Warn on large files, provide progress reporting

### Risk 3: Cross-platform path issues
**Mitigation:** Use `java.nio.file.Path` for platform-independent handling

### Risk 4: Symlink confusion
**Mitigation:** Follow symlinks, copy actual file content

## Open Questions

1. Should we validate image file formats (PNG, JPG, SVG only)? **Decision: No, trust user**
2. Should we warn on absolute paths (not portable)? **Decision: Yes, warn but allow**
3. Should we support glob patterns (`images/*.png`)? **Decision: No, future enhancement**
4. Should we create a manifest file listing all assets? **Decision: No, not needed**
