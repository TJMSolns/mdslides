# v0.4.0 Release Ceremony Document

**Version:** v0.4.0
**Release Date:** TBD (in development)

## Executive Summary

v0.4.0 introduces three major features to improve theme management and user experience:

1. **Directory-Based Themes** - Themes become self-contained directories with assets
2. **Background Images** - Per-slide and template-specific background image support
3. **Configuration Management** - CLI-based configuration system with three-tier precedence

**Breaking Changes:** Yes - themes must migrate from single JSON files to directory structure

**Migration Path:** Manual migration (documented in release notes)

---

## Committed Features

### US-016: Directory-Based Themes [MUST]
**Priority:** MUST (foundation for US-011, US-012)
**Status:** 🟢 Event Storming Complete

**Summary:**
Themes transition from single JSON files to self-contained directories containing `theme.json` plus assets (backgrounds, logos, etc.). This enables portable, shareable themes with clear path resolution.

**Acceptance Criteria:**
- ✅ Theme structure: `themes/retisio/theme.json` + assets in subdirectories
- ✅ Paths in theme.json are relative to theme directory
- ✅ CLI discovers themes in configured theme directory
- ✅ Theme images copied to output directory (preserving structure)
- ✅ Clear error messages when theme not found

**Key Decisions:**
- Clean break from v0.3.0 (no backward compatibility)
- Theme-relative path resolution
- Default theme directory: `./themes/`
- Configurable via configuration system (US-017)

**Related Governance:**
- [PDR-013: Directory-Based Theme Architecture](../governance/pdr/PDR-013-directory-based-theme-architecture.md)
- [Event Storming: US-016](../planning/event-storming-US-016.md)

### US-011: Per-Slide Background Images [SHOULD]
**Priority:** SHOULD
**Status:** 🟢 Event Storming Complete
**Depends On:** US-016

**Summary:**
Authors can override theme/template backgrounds on individual slides using frontmatter. Supports both simple string syntax and extended object syntax with opacity.

**Acceptance Criteria:**
- ✅ Slide frontmatter accepts `background` field (string OR object)
- ✅ Simple form: `background: images/custom-bg.png`
- ✅ Extended form: `background: { image: "...", opacity: 0.3 }`
- ✅ Slide background overrides theme/template background
- ✅ Background images copied to output directory
- ✅ Validation warns if background image missing (graceful degradation)

**Key Decisions:**
- Background as slide metadata (not slot)
- Union type: String | BackgroundConfig
- Fallback chain: slide → template → theme → color
- Warn on missing images (don't fail build)

**Related Governance:**
- [PDR-011: Background Image Architecture](../governance/pdr/PDR-011-background-image-architecture.md)
- [Event Storming: US-011](../planning/event-storming-US-011.md)

### US-012: Template-Specific Background Defaults [SHOULD]
**Priority:** SHOULD
**Status:** 🟢 Event Storming Complete
**Depends On:** US-016

**Summary:**
Theme designers can specify different default backgrounds per template type. Title slides automatically get one background, content slides get another, etc.

**Acceptance Criteria:**
- ✅ Theme JSON accepts `templateBackgrounds` map (template name → image path)
- ✅ Background selection priority: slide override > template default > theme default
- ✅ Example: `{"title": "backgrounds/title.png", "content": "backgrounds/content.png"}`
- ✅ Template backgrounds copied to output directory
- ✅ Invalid template names in `templateBackgrounds` warned (not error)

**Key Decisions:**
- templateBackgrounds stored in Theme aggregate
- Simple string values (no per-template opacity in v0.4.0)
- Unknown template names warned but allowed (extensibility)
- Optional field (backward compatible with v0.3.0 themes)

**Related Governance:**
- [PDR-011: Background Image Architecture](../governance/pdr/PDR-011-background-image-architecture.md)
- [Event Storming: US-012](../planning/event-storming-US-012.md)

### US-017: Configuration Management CLI [SHOULD]
**Priority:** SHOULD
**Status:** 🟢 Event Storming Complete
**Depends On:** None (independent)

**Summary:**
Users can configure MDSlides defaults via CLI commands or interactive wizard. Supports three-tier configuration (project/user/system) with clear precedence.

**Acceptance Criteria:**
- ✅ `mdslides config` (no args) launches interactive configuration wizard
- ✅ `mdslides config set theme-dir /path` sets configuration non-interactively
- ✅ `mdslides config get theme-dir` retrieves configuration (for scripting)
- ✅ `mdslides config list` shows all settings with sources
- ✅ Three-tier precedence: project (`./.mdslides/config`) → user (`~/.mdslides/config`) → system
- ✅ Configuration keys: `theme_dir`, `default_theme`, `copy_images`, `default_output_dir`

**Key Decisions:**
- Simple key=value file format (not JSON/YAML)
- Dual-mode CLI: interactive wizard + non-interactive commands
- Default scope: user config (safest)
- Missing config files use defaults (not error)
- Validate at set time (fail fast)

**Related Governance:**
- [PDR-014: Configuration Management Strategy](../governance/pdr/PDR-014-configuration-management-strategy.md)
- [Event Storming: US-017](../planning/event-storming-US-017.md)

## Architecture Decisions

### Domain Model Changes

#### New Aggregates
- **Config** - Configuration settings with three-tier precedence

#### Enhanced Aggregates
- **Slide** - Add `backgroundImage: Option[SlideBackground]`
- **Theme** - Add `templateBackgrounds: Map[String, String]`

#### New Value Objects
- **SlideBackground** - Union type (String | BackgroundConfig)
- **BackgroundConfig** - Extended background config with opacity
- **ConfigSource** - Tracks config value source (Project/User/System/Default)

### Infrastructure Changes

#### New Components
- **ThemeLoader** - Load themes from directories
- **ConfigFile** - Read/write configuration files
- **ConfigCommand** - CLI configuration management
- **InteractiveWizard** - Interactive configuration flow

#### Enhanced Components
- **ThemeJsonAdapter** - Resolve theme-relative paths, parse templateBackgrounds
- **SlideParser** - Parse background field from frontmatter
- **ImageAssetCopier** - Extract and copy theme + slide + template backgrounds
- **HTMLRenderer** - Apply background fallback chain

### Key Principles

1. **Graceful Degradation** - Warn on missing images, don't fail build
2. **Clear Precedence** - Unambiguous fallback chains (slide → template → theme)
3. **Fail Fast** - Validate configuration at set time, not runtime
4. **Self-Contained Themes** - Themes as portable directories
5. **Backward Compatibility** - Where possible (templateBackgrounds optional)

## Migration Guide (for v0.3.0 Users)

### Breaking Changes

#### Theme Structure Changed
**Before (v0.3.0):**
```
themes/
  retisio.json
  minimal.json
```

**After (v0.4.0):**
```
themes/
  retisio/
    theme.json
    backgrounds/
      title-page.png
  minimal/
    theme.json
```

**Migration Steps:**
1. Create directory for each theme: `mkdir -p themes/retisio`
2. Move JSON file into directory: `mv themes/retisio.json themes/retisio/theme.json`
3. Move theme assets into theme directory
4. Update image paths in theme.json to be relative to theme directory

### New Features (Optional)

#### Per-Slide Backgrounds
Add `background` field to slide frontmatter:

```yaml
---
template: title
background: images/hero.png
---
```

Or with opacity:
```yaml
---
template: title
background:
  image: images/hero.png
  opacity: 0.7
---
```

#### Template-Specific Backgrounds
Add `templateBackgrounds` to theme.json:

```json
{
  "name": "My Theme",
  "templateBackgrounds": {
    "title": "backgrounds/title-page.png",
    "content": "backgrounds/content-page.png"
  }
}
```

#### Configuration Management
Set defaults to avoid repeating CLI flags:

```bash
# Interactive wizard
mdslides config

# Or non-interactive
mdslides config set theme-dir ~/.mdslides/themes
mdslides config set default-theme retisio
mdslides config set copy-images false
```
