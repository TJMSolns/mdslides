# v0.4.0 Architecture Decisions Summary

**Date:** 2024-12-26
**Status:** Ready for Event Storming
**Features:** Directory-Based Themes, Background Images, Configuration Management

---

## Decision Log

### 1. Theme Structure: Directory-Based (Clean Break)
**Decision:** Themes are directories containing `theme.json` + assets
**Rationale:**
- Self-contained, portable theme packages
- Aligns with slide deck directory structure
- Clearer path resolution for theme assets

**Migration:** Clean break - no backward compatibility for single-file themes
- `themes/retisio.json` → `themes/retisio/theme.json`
- Existing themes must be migrated

**Structure:**
```
themes/retisio/
  theme.json
  backgrounds/
    title.png
    content.png
  logos/
    retisio-logo.png
```

---

### 2. Image Copying: Unified Behavior
**Decision:** All images (theme + content) follow same copying rules

**Behavior:**
- **Default:** Copy all local images to output directory
- **Remote URLs:** Download and copy to output directory
- **`--no-copy-images`:** Disable ALL copying (theme + content)

**Rationale:**
- "A slide deck should have all images in one place"
- Consistent mental model
- Self-contained output by default
- One flag controls everything

---

### 3. Theme Discovery: Configurable Search Path
**Decision:** Single configurable theme directory (not PATH-style)

**Configuration Locations (precedence order):**
1. `./.mdslides/config` (project-local, highest priority)
2. `~/.mdslides/config` (user-level)
3. `/usr/local/etc/mdslides/config` (system-level, lowest priority)

**Config Format:**
```
theme_dir=~/.mdslides/themes
```

**CLI Management:**
- `mdslides config` (no args) → Interactive configuration wizard
- `mdslides config set theme-dir /path` → Non-interactive set
- `mdslides config get theme-dir` → Non-interactive get (for scripting)
- `mdslides config list` → Show all settings

**Rationale:**
- Simple single source (not PATH complexity)
- Supports both interactive and scripting workflows
- Standard Unix config locations

---

### 4. Per-Slide Background Override: String OR Object
**Decision:** Support both simple string and extended object syntax

**Simple Form (90% use case):**
```markdown
---
template: content
background: images/custom-bg.png
---
```

**Extended Form (future-proof):**
```markdown
---
template: content
background:
  image: images/custom-bg.png
  opacity: 0.3
---
```

**Rationale:**
- Simple form for common case (no friction)
- Object form for future extensions (opacity, position, blend modes)
- YAML parsers handle both naturally
- Backward compatible migration path

---

### 5. Template-Specific Backgrounds: Theme Ownership
**Decision:** Template → background mapping lives in Theme aggregate

**Theme JSON:**
```json
{
  "name": "Retisio",
  "background": {
    "color": "#FFFFFF",
    "image": "backgrounds/default.png"
  },
  "templateBackgrounds": {
    "title": "backgrounds/title-page.png",
    "content": "backgrounds/content-page.png",
    "diagram": "backgrounds/diagram-page.png"
  }
}
```

**Fallback Chain:**
1. Slide.backgroundImage (highest priority - user override)
2. Theme.templateBackgrounds[templateName] (template-specific default)
3. Theme.background.image (theme default)
4. Theme.background.color (always present - fallback)

**Rationale:**
- Themes define styling, Templates define structure
- Different themes can style same template differently
- Aligns with DDD aggregate boundaries

---

### 6. Slide Background: Metadata (Not Slot)
**Decision:** Background is slide-level metadata, not content

**Domain Model:**
```scala
// CURRENT
case class Slide(
  id: SlideId,
  templateName: String,
  slots: Map[String, String]
)

// PROPOSED
case class Slide(
  id: SlideId,
  templateName: String,
  slots: Map[String, String],
  backgroundImage: Option[SlideBackground] = None  // NEW
)

// Support both string and object
type SlideBackground = Either[String, BackgroundConfig]

case class BackgroundConfig(
  image: String,
  opacity: Option[Double] = None
  // Future: position, blendMode, etc.
)
```

**Rationale:**
- Background is presentation property, not content
- Slots are for user-written text/content
- Matches how templateName works (metadata)

---

### 7. Theme Domain Model Changes
**Decision:** Add templateBackgrounds to Theme aggregate

**Domain Model:**
```scala
// CURRENT
case class Theme(
  name: String,
  version: String,
  background: Background,
  colors: ColorScheme,
  fonts: FontScheme,
  spacing: Spacing,
  syntax: SyntaxColors,
  slideCounter: SlideCounter
)

// PROPOSED
case class Theme(
  name: String,
  version: String,
  background: Background,
  templateBackgrounds: Map[String, String] = Map.empty,  // NEW
  colors: ColorScheme,
  fonts: FontScheme,
  spacing: Spacing,
  syntax: SyntaxColors,
  slideCounter: SlideCounter
)
```

---

### 8. Validation Strategy: Early Warnings (Don't Fail)
**Decision:** Validate images at validation phase, warn but don't fail build

**Behavior:**
- Missing local images → Warning (continue rendering)
- Invalid URLs → Warning (continue rendering)
- Graceful degradation to fallback background

**Rationale:**
- Consistent with US-006 image validation
- Allows partial builds during development
- Warnings appear early (before render)

---

### 9. Output Directory Structure: Preserve Source Structure
**Decision:** Maintain directory structure from source in output

**Output Layout:**
```
output/
  presentation.html
  backgrounds/        ← from theme (themes/retisio/backgrounds/)
    title.png
    content.png
  images/             ← from slides (my-slides/images/)
    diagram.png
    logo.png
```

**Collision Handling:**
- Different directories → No collision
- Same directory + same filename → Warn and skip duplicate

**Rationale:**
- No renaming needed (HTML references match source)
- Existing US-006 code already does this
- Matches user's mental model of file organization
- Simpler than flattening with collision resolution

---

### 10. Logo Support: Deferred to v0.5.0+
**Decision:** Logos OUT OF SCOPE for v0.4.0

**Rationale:**
- Focus on background images first
- Logo positioning needs more design work
- Content images can serve as interim solution
- Avoid feature creep in v0.4.0
