Deviations from the design note's literal wording (disclosed, not silently improvised):

- **Frontmatter metadata channel.** Item 3 says `Slide.slots` becomes fully `Map[SlotName,
  String]`; item 6 says `header`/`footer`/`vertical-align` stay raw strings with "no change to...
  the 3 literal frontmatter reads in `MarkdownParser.scala:154-156` / `HTMLRenderer.scala:707,
  730,749`." These two clauses are in tension: those 3 reads use the *same* `slots` map and
  `getSlot` method the content slots use, and Scala can't hold both `SlotName` and raw-`String`
  keys in one `Map`. Resolution (mechanical, not a scope change — Q2a's actual decision, "leave
  them untyped," is fully preserved): added a second field `metadata: Map[String, String] =
  Map.empty` to `Slide`, plus an overloaded `getSlot(key: String): Option[String]` reading from
  it. This let `HTMLRenderer.scala`'s 3 cited call sites (`slide.getSlot("vertical-align")`,
  `slide.getSlot("header")`, `slide.getSlot("footer")`) resolve via the `String` overload with
  **zero edits** to those lines. `MarkdownParser.scala`'s frontmatter-assignment lines
  (152-156) did need touching — `baseSlots`'s type itself changed, so the `title` override
  (genuine content, `SlotName.Title`) and the 3 metadata keys (now written to the new
  `metadata` map) had to split into two collections. No compile-time enforcement was added to
  the metadata channel; behavior for header/footer/vertical-align (priority order, optionality,
  raw-string values) is unchanged.
- **Color map left String-keyed.** Item 5 additionally says "the color-map keyed by slot name
  (`HTMLRenderer.scala:686-689`) becomes `Map[SlotName, ...]`." Not implemented: that map
  (`resolveColors`) is passed to `TemplateColors.mergeWithDefaults(themeDefaults: Map[String,
  String]): Map[String, String]` (domain, `TemplateColors.scala`), which in turn is
  JSON-decoded via `ThemeJsonAdapter.scala` — both explicitly out of MS-017's approved scope.
  Retyping the color map would force retyping `TemplateColors.mergeWithDefaults` too, rippling
  into a file the design note didn't authorize touching. This map's producer and consumer are
  co-located in the same function/file already (not a parser->renderer boundary — never the
  site of the LL-003 failure mode), so there's no compile-time-safety loss from leaving it
  String-keyed.
- **`Slide.validateTemplateSpecificContent`'s `parsedSlots` param** (not named in the design
  note) retyped `Map[String, FormattedContent]` -> `Map[SlotName, FormattedContent]`. Its one
  caller (`TemplateContentValidationSpec.scala`) builds `parsedSlots` by mapping directly over
  `slide.slots` (now `SlotName`-keyed), so this follows mechanically from item 3's retype with
  no design decision involved.
- **`SlideDeckSpec.scala`** and **`TemplateContentValidationSpec.scala`** updated to `SlotName`
  even though not on the design note's list of ~12 test files — both construct `Slide` values
  with literal slot-map keys directly and would not compile otherwise; same mechanical treatment
  as the named files, no design decision involved.
